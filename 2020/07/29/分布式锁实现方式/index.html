高并发相关技术

分布式锁

在单机中可以通过语法提供的关键字实现锁，保证数据安全。如java中使用synchronized、lock等。但是在分布式系统中，如何保证多个主机之间保持数据安全，就需要分布式锁的设计。

Redis实现分布式锁

对于一个锁，要满足三个条件：互斥性、多进程可见、可重入。

互斥性

使用redis的setnx（set if not exist）命令setnx key value，在redis中添加一条记录。

setnx只有在第一次写入时返回1（true），其他情况下返回0（false）。

为了产生死锁，需要给记录设置一个存活时间，到期删除，使用setex命令即可。

问题：如果一个服务刚刚设置了记录，尚未设置超时时间时系统宕机了，可能会存在死锁问题。

解决：使用set key value nx ex timeout统一添加记录和设置过期时间。

问题：如果进程可以删除别的进程设置的记录，就有可能在释放锁时释放别人的锁，导致多个进程同时执行，违背了互斥性

解决：在设置锁时加入自己线程的唯一标识，删除锁前，判断该锁是否是自己的，不是则不删除。

多进程可见

redis本身就是另外的进程，天然满足条件。

可重入

可重入的概念就是，在同一个线程中，如果在外层获得了锁，内层仍可获得该锁，避免无谓的释放-获取锁的过程。同时也减少死锁产生的概率。

个人理解：定性 ——-> 定量

尝试获取锁，如果失败则判断该锁是否是自己的，如果是则允许再次获取锁。同时要记录获取的次数。

释放锁的过程，不能直接删除，而是将重入次数减一，当次数为0时删除锁。

如果想要动态调整一个客户端获取锁的时间，可以设置一个看门狗，设置一个扫描时间，如10s，每隔10s查看是否客户端仍然拥有锁，如果是，则延长持有锁的时间。

使用lua脚本保持操作的原子性。

redis实现分布式锁的最大问题

面对redis集群时，设置锁是在master中设置的，由主从异步复制同步给salve相关信息。如果master宕机，就会有salve转为master，然后可能存在另外的客户端成功加锁，破坏了互斥性，进而导致各种脏数据的产生。所以缺陷就是在redis master实例宕机的时候，可能导致多个客户端同时完成加锁。

个人理解：放弃了对“资源”的实际掌握，而是采用逻辑上的控制手段：对于需要加锁的资源（代码块），在执行之前判断能不能在redis中”注册自己“，如果不成功，则认为获取锁失败（即存在另一个主机在redis中注册了自己）。
