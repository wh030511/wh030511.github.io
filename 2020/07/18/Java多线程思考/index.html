Java多线程

关于Java多线程的复习与思考。

首先介绍两个概念：

  • 并行：在同一时刻发生在不同实体上的多个事件。
  • 并发：在同一时间间隔发生在同一实体上的多个事件。

得益于硬件的发展，如今的PC可以拥有多核，每个处理器的性能也十分强大。在这种情况下，充分利用硬件的性能来提高软件的使用体验就是一个水到渠成的需求了。

每个核去独立的处理进程就是并行，单个核处理进程造成的“伪并行”的现象就是并发。没错，在我的认知中，由于计算机CPU每秒可以进行数十亿次的浮点计算，所以在宏观层面，并发和并行并无区别。我们也乐于如此，尽管理性告诉我们只有一个勤勤恳恳的管家在做事，只不过他做的太快了以至于让我们误以为有许多仆人。但是管他呢，一边听音乐一边浏览网页不香吗？

这个管家实在太厉害了，于是他分成了许多小管家，每个进程都由一个小管家去管理，这些小管家也掌握了“分身”的技能，这就是多线程了。

坦率的讲，小管家并不高兴，他并不觉得自己做了更多的事：我本可以一鼓作气的完成我的工作，为什么要我东一榔头西一棒子去做同一件事呢？来回的折腾甚至还更耗时了呢！

小管家忘了，不是所有人都能像他这样厉害哒！那边蠢蠢的IO还在哼哧哼哧的给小管家运来需要的数据呢，如果是单线程的话，做不了多久小管家就要停下来等等他愚蠢的IO朋友。而黑心的主人一点也不想让小管家休息，怎么办呢？有啦！主人把这个任务分成好几个小任务，小管家做完小任务A就去做小任务B吧，等小任务B做完了，小管家的IO朋友也运来需要的数据了，要是这样小管家还能休息，就拆出更多的任务C、任务D来，嘿嘿嘿。黑心的主人得意的笑了出来。

所以，多线程没能提高小管家的效率，但是提高了小管家的占用率，变相的提高了整个任务的效率。

Java多线程环境中，Java给每个线程一个自己独立的栈空间和程序计数器，栈帧中存放局部变量、指令等信息，程序计数器就指明了下一次要执行的指令，这样，每个线程就可以独立的进行工作了。

与进程不同的是，线程之间共享内存空间（堆和方法区），不像进程并发那样哪怕不进行通信也能独立的完成自己的任务。所以多线程天然的就具有通信的条件：所谓通信，不就是交换数据吗？

但是，共享了数据区，各个线程如何保证自己的数据不被别的线程恶意篡改呢？这就是多线程的线程安全问题。

线程的安全

线程的安全问题，主要可以通过以下几种思路解决。

一、藏起来

既然放在堆里不靠谱，所有线程都能访问，那么就把他放到每个线程各自的栈空间不就得了？

将对象的成员变量编程run方法的局部变量，不得不说，确实解决了线程的安全问题。但是无异于鸵鸟，原本天然的通信条件也被破坏了，并不是一个靠谱的选择。

二、人人有份

既然这个数据大家都要用，不妨每人发一个，大家都操作自己的，岂不美哉？

一种空间换时间的策略。Java的ThreadLocal类就是用这种思想来解决线程安全问题的。

ThreadLocal类中有一个成员变量ThreadLocal.ThreadLocalMap，是ThreadLocal的一个静态内部类，其中维护了一个Entry数组用来存放数据，key是当前的线程，value是存放的值。

通过ThreadLocal的set和get方法来改变或者获取当前线程的value。
