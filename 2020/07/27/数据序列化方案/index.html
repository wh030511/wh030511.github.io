序列化方案

网络通信和通用数据交换技术，大体介绍xml、json、protobuf、thrift四种方案。

XML

定义：可扩展标记语言，使用DOM树的形式描述数据，可以用于数据存储和传输。

优点：独立于语言和平台，发展较早，使用范围广；格式统一，可以与其他系统远程交互。

缺点：数据结构太过笨重，在网络传输过程中存在许多无谓的消耗。不同的浏览器之间解析方式不同，故需要编写许多重复代码。其组织也比较复杂，各种子节点父节点，解析困难。

Json（JavaScript Object Notation：js对象表示法）

定义：轻量级数据交换格式，独立于语言和平台。

优点：javascript原生支持json，解析速度很快；json格式简单，传输成本比xml低很多；服务器解析速度快。

缺点：安全性不够高，没有错误处理。

Protocol Buffer

pb是一种语言无关、平台无关、可扩展的序列化结构数据的方法，它可用于（数据）通信协议、数据存储等。由于是二进制传输方案，所以性能极强。同时，拥有很好的扩展性和兼容性。

使用步骤：

1. 创建.proto文件，定义数据结构：语法可以百度，不难。

2. 使用protoc编译.proto文件生成读写接口：提供将数据结构序列化和反序列化的方法。针对不同的语言（可以在编译时指定），可以生成相应语言的输入输出流，用于磁盘IO。

3. 调用提供的接口，实现序列化、反序列化和读写

编码方案

采用TV或者TLV的方式进行编码，取决于Tag的后三个bit规定的wire_type，取值如下：

Type Meaning Used For
0 Varint int32/int64/uint32/uint64/sint32/sint64/bool/enum
1 64-bit fixed64/sfixed64/double
2 Lentgh-delimited String/bytes/embedded/message/package/repeated/fields
5 32-bit fixed32/sfixed32/float

(注：原3、4的start group和end group已被废弃)

Varints编码：

规则：

  1. bit表示是否是最后一个字节，参照utf-8编码格式。
  2. 存储补码
  3. 补码低位在前（为编码实现的一个优化）

优点：对于小数来说，会减少很多空间的消耗，而小数字在传输的过程中是大概率事件。

缺点：大数和负数反而会浪费空间

解决：对于大数，采用64-bit或者32-bit来定义长度，对于负数采用ZigZag编码，即将有符号整数映射到无符号整数，然后再使用 Varints 编码。

Length-delimited类型

即在message中使用之前定义过的message，采用Tag-Length-Value形式编码。

packed编码

在proto3中默认置为true，就是把 repeated primitive 的编码结果打包，从而进一步压缩空间，进一步提高效率、速度。

打包的含义其实就是：原先的 repeated 字段的编码结构为 Tag-Length-Value-Tag-Length-Value-Tag-Length-Value…**，因为这些 Tag 都是相同的（同一字段），因此可以将这些字段的 Value 打包，即将编码结构变为 **Tag-Length-Value-Value-Value…

Thrift

thrift即是一个rpc框架，也可以作为一个序列化方案，提供一站式解决方案。

更应该和grpc对比。

thrift通过一个中间语言IDL来定义rpc的数据类型和接口（在.thrift文件中），然后通过特殊的编译器生成不同语言的代码，生成的代码中不但包含目标语言的接口定义，方法，数据类型，还包含有rpc协议层和传输层的实现代码。
